# День 3 — полный цикл проверки (код-тесты + smoke через Playwright MCP)

Требует, чтобы текущая сессия была запущена с рабочей директорией
`C:\Projects\CaptainBlood` — Playwright MCP подключается на уровне проекта
(`.mcp.json`) и виден только при старте сессии в этой папке.

## Шаг 1 — код-тесты (Уровень 1)

```
./gradlew test --console=plain
```

Разобрать вывод: сколько тестов, сколько упало. При падении — открыть конкретный
класс теста, найти причину (не переписывать тест "чтобы прошёл", если падение
вскрывает реальный баг в бизнес-логике — сообщить об этом отдельно).

## Шаг 2 — поднять сервис локально

1. Проверить, что локальная Ollama запущена (`curl http://localhost:11434/api/tags`).
2. Запустить сервис: `./gradlew run` → в интерактивном меню выбрать пункт запуска
   HTTP-сервиса (см. `Main.kt`, `runServerMode`) — слушает `http://localhost:8080`.
3. Логин/пароль для Basic Auth — читать из `src/main/resources/config.properties`
   (`PRIVATE_LLM_LOGIN` / `PRIVATE_LLM_PASSWORD`), **никогда не печатать пароль в
   отчёт/скриншоты/лог** — только использовать при вводе в форму.

## Шаг 3 — smoke-сценарии (Уровень 2)

Сценарии — `docs/AdvancedCourse_Day3_SmokeScenarios.md` (5 штук: health-check без
авторизации, успешный вход + ответ по теме, отказ при неверном пароле, rate limit,
отказ на вопрос не по теме).

Прогонять **строго против `http://localhost:8080`**, никогда против боевого
`zhdun.duckdns.org` (им пользуется реальный ребёнок).

Для каждого сценария через Playwright MCP:
- открыть страницу/выполнить действие по шагам сценария,
- сделать скриншот на каждом значимом шаге (сохранять в `docs/day3_screenshots/`,
  эта папка — в `.gitignore`, скриншоты не коммитить, только приложить к отчёту/видео),
- зафиксировать факт: совпал ли результат с ожидаемым.

## Шаг 4 — единый отчёт

Собрать markdown с разделами:
1. Итог Уровня 1 — сколько тестов, все прошли/список упавших.
2. Итог Уровня 2 — по каждому из 5 сценариев: пройден/не пройден + ссылка на скриншот.
3. Если что-то упало на любом уровне — отдельный блок "Проблема": в каком месте
   (файл/сценарий/шаг), гипотеза причины, не чинить молча вне явного запроса.

## Когда перезапускать flow

Триггер "я задеплоил новую фичу — обнови smoke и прогони всё заново":
1. Перечитать `docs/AdvancedCourse_Day3_SmokeScenarios.md` — если фича меняет
   UI/API (новый эндпоинт, новое поведение чата) — добавить/поправить сценарий.
2. Повторить Шаги 1–4 целиком.
