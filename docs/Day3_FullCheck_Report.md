## Проблема 1 — методологический артефакт тестовой оснастки (не баг сервиса)

**Где:** первая попытка сценария 2 (навигация через
`http://login:pass@localhost:8080/`).

**Симптом:** после входа в чате на любой вопрос выпадало «Не докричаться
до корабля — проверь соединение и попробуй снова.», баннер статуса
показывал «нет связи с кораблём».

**Причина (подтверждена):** Chrome, управляемый Playwright через
`page.goto()` с credentials, встроенными в URL, не убирает их из
`location.href` — относительные `fetch(...)` в JS страницы после этого
падают по спецификации Fetch: `TypeError: Failed to execute 'fetch' on
'Window': Request cannot be constructed from a URL that includes
credentials`. Прямой `fetch('/chat')` из консоли после повторной навигации
на URL БЕЗ креда (Chrome сам переиспользовал закэшированный Basic Auth)
отработал штатно, статус вернулся в «на связи» — сам сервис не при чём.

**Обход в тесте:** после первого успешного входа с кредом в URL —
повторная навигация на чистый `http://localhost:8080/` (без креда), Chrome
переиспользует закэшированную Basic Auth-сессию, `fetch()` из JS страницы
работает нормально.

**Не чинить в коде сервиса** — реальный пользователь логинится через
нативный диалог браузера, где `location.href` креды не содержит, этот
кейс его не касается.

## Проблема 2 — сценарий 5 недетерминирован, нет keyword-gate для офтопика (реальный пробел)

**Где:** `Main.kt` — системный промпт `VPS_ANSWER_SYSTEM_BASE` (строки
163-190) и `answerWithRag` (строка 1223).

**Симптом:** три попытки офтопик-вопроса в этом прогоне дали три разных
результата:
1. «Как приготовить пиццу?» (1-я попытка, до перезапуска сервиса) — капитан
   дал ПОЛНЫЙ рецепт пиццы, без единого намёка на увод в морскую тему
   (скриншот с этой попытки случайно удалён при уборке — см. ниже).
2. «Как приготовить пиццу?» (2-я попытка, после перезапуска, тот же
   вопрос) — сорвалось в языковой дрейф модели, ответ — техническая
   заглушка «Не знаю. Ответ модели содержал недопустимые символы...»
   (`OllamaClient.kt:293`, после исчерпания `MAX_LANGUAGE_DRIFT_RETRIES`).
   Скриншот: `scenario5_offtopic_pizza.png`.
3. «Расскажи про футбол» — единственная из трёх попыток отработала как
   ожидается: капитан не дал фактов про футбол, мягко увёл в морскую тему
   («поговорить о море... на море играли в другой мяч»). Скриншот:
   `scenario5_offtopic_football.png`.

**Причина (подтверждена чтением кода, не гипотеза):** в
`VPS_ANSWER_SYSTEM_BASE` НЕТ ни одного правила про удержание темы
«море/пираты» — единственное близкое правило запрещает только пугающие/
жестокие/недетские темы («Никогда не затрагивай пугающие, жестокие или
недетские темы — если вопрос об этом, мягко переведи разговор...»,
`Main.kt:185-186`), оно не покрывает нейтральный офтопик (еда, спорт).
В `answerWithRag` нет отдельного keyword-gate/детерминированной ветки для
"вопрос не про море" — в отличие от безопасности/такелажа/акул/оружия/
истории (пункт 2 паттернов проекта в `CLAUDE.md`), для офтопика такого
гейта просто не существует. Из-за этого результат целиком зависит от
того, как конкретная генерация модели (`qwen2.5:7b-instruct`) сама
решит интерпретировать нейтральный вопрос — иногда отвечает по существу
(нарушение темы сервиса), иногда корректно уводит, иногда срывается в
языковой дрейф на неоднозначном промпте (известная особенность семейства
qwen2.5, см. `~/.claude/rules/SETUP_ENVIRONMENT.md`).

**Не чинил** — по правилу проекта сообщаю находку, не правлю код без
явного запроса. Похожий паттерн уже применён в проекте для безопасности/
такелажа/акул и т.д. (`VPS_SAFETY_KEYWORDS` и аналоги) — по аналогии
для сценария 5 напрашивается детерминированный gate вида
"вопрос не по морской/пиратской теме → отдельная жёсткая инструкция в
системном промпте", а не текущая надежда на общее правило "мягко
переведи", которое покрывает только пугающие темы.
