# Advanced-курс, День 3 — Smoke-сценарии (Уровень 2, Playwright MCP)

Пять пользовательских сценариев для UI-сервиса «капитан Блад» (`GET /`, `POST /chat`,
`GET /health`). Прогоняются агентом самостоятельно через Playwright MCP — открывает
браузер, кликает/печатает, проверяет результат, делает скриншот на каждом шаге.

Цель сервиса — приватный эрудит-бот про море/пиратов для одного ребёнка, поэтому
классического CRUD («создать сущность → проверить → удалить») здесь нет — сценарии
адаптированы под реальный функционал: чат + авторизация + защита от нагрузки.

**Прогон — ТОЛЬКО против локально поднятого экземпляра** (`./gradlew run`, локальная
Ollama), никогда не против боевого VPS (`zhdun.duckdns.org`) — им пользуется реальный
ребёнок, лишний трафик и особенно сценарий 4 (rate limit) не должны его трогать.

Логин/пароль берутся из `src/main/resources/config.properties` (в `.gitignore`,
не хардкодить в сценариях/отчётах/скриншотах).

---

## Сценарий 1 — health-check без авторизации

1. Открыть `GET /health`.
2. Ожидаемо: страница отдаётся БЕЗ запроса логина/пароля (в отличие от `/` и `/chat`).
3. Ожидаемо: тело ответа — JSON со статусом `ok` (доступность локальной Ollama).
4. Скриншот: содержимое страницы.

## Сценарий 2 — успешный вход и ответ на вопрос по теме

1. Открыть `GET /` — браузер должен запросить HTTP Basic Auth.
2. Ввести верные логин/пароль (из локального `config.properties`).
3. Ожидаемо: открывается HTML-морда чата (стилизация под судовой журнал).
4. Ввести вопрос по теме, например «Что такое коралловый риф?», отправить.
5. Ожидаемо: в течение разумного времени (см. `VPS_EXPECTED_ANSWER_SECONDS`) в чате
   появляется ответ, обратный отсчёт в UI перед этим не завис.
6. Скриншот: страница после входа, скриншот: ответ в чате.

## Сценарий 3 — отказ при неверном пароле

1. Открыть `GET /`, ввести НЕверный пароль.
2. Ожидаемо: браузер получает 401 и повторно показывает диалог Basic Auth (либо
   стандартную страницу ошибки браузера) — до HTML-морды и тем более до `/chat`
   доступ не проходит.
3. Скриншот: экран отказа/повторного запроса авторизации.

## Сценарий 4 — превышение rate limit

1. Войти корректно (см. Сценарий 2).
2. Отправить `VPS_RATE_LIMIT_MAX + 1` сообщений подряд быстро один за другим
   (значение константы — см. `Main.kt`, сейчас 5 → 6-е сообщение должно быть отклонено).
3. Ожидаемо: на последнем сообщении в чате появляется текст про превышение лимита
   запросов, а не обычный ответ модели.
4. Скриншот: сообщение об ограничении в чате.

## Сценарий 5 — отказ на вопрос не по теме (keyword-gate)

1. Войти корректно.
2. Задать вопрос, не относящийся к морской тематике, например
   «как приготовить пиццу» или «расскажи про футбол».
3. Ожидаемо: ответ — отказ/переключение на тему сервиса (эрудит про море), а не
   попытка ответить на посторонний вопрос.
4. Скриншот: ответ-отказ в чате.

---

## Формат отчёта

После прогона агент фиксирует по каждому сценарию:
- пройден / не пройден,
- путь к скриншоту(ам) шага,
- если не пройден — в каком именно шаге разошлось ожидание/факт, гипотеза причины.

Отчёт объединяется с результатом Уровня 1 (`./gradlew test`) в один markdown-файл
(см. `.claude/skills/day3-full-check/SKILL.md` — единая точка запуска обоих уровней).
