# Сравнение LLM-моделей для ответа капитана Блада

Живой журнал сравнения кандидатов на роль `VPS_MODEL` (`Main.kt`) — модели,
которая генерирует ответы чата на VPS (2 vCPU, 8 ГБ RAM, без GPU). Дополнять
эту таблицу при появлении новых моделей, не создавать новый файл.

## Методология (обязательно соблюдать при добавлении новых строк)

1. **Качество — сначала локально** (машина разработчика, GPU-Ollama). От
   железа не зависит: веса и промпт одинаковы что на CPU, что на GPU, только
   скорость разная. Не тестировать качество на VPS — это тратит время и риск
   для боевой машины впустую (см. `feedback-test-locally-before-vps-deploy`
   в памяти Claude — эта ошибка повторялась дважды за 2026-07-13/07-21).
2. **Скорость — только на VPS**, и только для уже одобренного по качеству
   кандидата, одним финальным прогоном. Локальный GPU даёт нерепрезентативные
   цифры (кандидаты `vikhr`/`ministral`/`phi-4-mini`/`qwen2.5:1.5b` ниже
   замерены ЛОКАЛЬНО на GPU только для быстрой проверки качества — их время
   в таблице НЕ сравнимо с VPS-колонкой напрямую).
3. **Между вызовами разных моделей на VPS — обязательно выгружать** через
   `"keep_alive": 0` в запросе к Ollama (или `ollama stop <model>`). Причина:
   2026-07-21 `ollama.service` упал от OOM-killer на VPS с 7.8 ГБ RAM, когда
   несколько 3-4B моделей одновременно остались резидентными в памяти между
   тестами — сервис поднялся сам за секунды (`systemd Restart=always`), но
   риск реален и легко избегается выгрузкой.
4. **Adversarial-тест на путаницу сущностей** (`composite_sharks`) —
   обязательная проверка для любого кандидата ≤4B. Вопрос: «Расскажи про
   самых опасных акул» с контекстом из 2-3 разных видов акул одновременно
   (белая/тигровая/акула-бык). Малые модели систематически путают, какой
   факт к какому виду относится (см. историю бага ниже) — это не гипотетика,
   это единственный найденный класс ошибки, который отсеял 5 из 7 кандидатов
   ниже. Промпт-файлы для повтора теста — в scratchpad сессии 2026-07-21
   (`bench_prompts_composite.json`, `bench_prompts_fix.json`), при следующей
   проверке нужно будет пересобрать аналогично (файлы во временной папке, не
   в репозитории).

## Текущее решение (2026-07-21, финал дня)

**`qwen2.5:7b-instruct`** — после того же дня, полного с `3b`, галлюцинации
победили экономию времени. Хронология в рамках одного дня:

1. Переключено на `3b` + промпт-правило `VPS_SHARK_ENTITY_ANCHOR_RULE` —
   среднее время ответа ~38-66с (было 180с+ на `7b`), adversarial-тест на
   составные вопросы про акул пройден чисто, задеплоено.
2. **На живом трафике** (не в синтетических тестах) `3b` дважды
   галлюцинировала за пределы того, что чинил фикс: придумала
   несуществующий вид коралла («Белый Барсик», затем «Акапулько-коралл» —
   город в Мексике, даже не в Карибском море), и перепутала гальюн
   (носовой корабельный туалет) с боевой надстройкой на корме.
3. Усиление системного промпта (явный запрет выдумывать виды
   животных/растений + правило «честно скажи "не знаю"» для вопросов
   «какой самый X») **не помогло** — `3b` всё равно придумала другое
   название коралла. `7b` на тех же контекстах ответила корректно.
4. **Вывод**: промпт-фиксы лечат ТОЧЕЧНЫЕ баги (shark anchor rule
   сработал), но не компенсируют общий разрыв в надёжности следования
   инструкциям между 3B и 7B — это предел возможностей модели, не
   формулировки промпта.
5. Решение — вернуться на `7b` (~3 минуты на ответ), и вместо борьбы за
   скорость модели добавить **обратный отсчёт в UI**
   (`VPS_EXPECTED_ANSWER_SECONDS`, `vpsIndexHtml`/JS) — честно показывать
   ребёнку ожидаемое время ответа вместо голой анимации "печатает...".
   При смене модели в будущем — поправить и это значение.

Живой сквозной тест через реальный `/chat`-эндпоинт подтверждён на обеих
моделях (`3b` — на составном вопросе про акул при 66с; `7b` — на вопросе про
кораллы, честно ответила «не знаю»).

## Проверенные и отклонённые направления (не тестировать повторно без новой причины)

- **Платный облачный роутер** (`routerai.ru`, уже подключён в `client/LlmClient.kt`
  для watchdog) — экономически незначим для этого масштаба (~0.02₽/ответ), но
  сознательно отклонён: цель проекта — обкатать паттерн полностью локального
  хранения/обработки данных (применимо к конфиденциальной информации в
  организации, где облако не вариант в принципе), не оптимизация стоимости.
- **"Открытые" модели известных брендов** (Grok/xAI, GLM/z.ai, GPT-oss/OpenAI,
  Claude/Anthropic) — либо не существуют в форме открытых весов (Anthropic),
  либо существуют, но требуют ≥16-24 ГБ RAM/VRAM даже в минимальном варианте
  (GPT-oss:20b, GLM-5.2) — не проходят по железу VPS (7.8 ГБ RAM, без GPU).
  Единственное исключение — Google Gemma (см. `gemma3:4b` в таблице).
- **Русскоязычные файнтюны крупнее 1B** (`Vikhr-llama3.1:8b`, `Saiga`,
  `vikhr_nemo:12b`) — по размеру не отличаются от уже отклонённого `7b`
  (тот же порядок скорости), не тестировались специально ради этого.
- **Cohere Aya** — минимальный размер на Ollama 8B (`aya:8b`), не 3.35B
  ("Tiny Aya" из маркетинга Cohere не опубликована на Ollama на 2026-07-21).
- **IBM Granite 4.1** — русский язык прямо не входит в официально
  поддерживаемый список языков модели, не тестировалась по этой причине.
- **Falcon3** — минимальный официальный размер 1B, тот же класс, что уже
  провалившийся `vikhr`/`deepseek-r1` на 1-1.5B.
